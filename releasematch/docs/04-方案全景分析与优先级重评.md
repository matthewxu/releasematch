## 一、方案本质：这不是「下载站」，而是「Release 导航站」

整个方案的战略定位可以概括为三层：

```
┌─────────────────────────────────────────────────────────────┐
│  流量层（SEO）— 内容轨 C0~C4                                 │
│  独立域名 + 冷启动分批 + Information Gain 页面               │
│  目标：M12 日 UV 500~1,200，5K~10K 页                        │
├─────────────────────────────────────────────────────────────┤
│  差异化层（IG）— 工具轨 T1~T3                                │
│  Recommended Release + 对版分析 + 测速 + Group 信誉          │
│  目标：每页提供竞品页没有的「增量信息」                       │
├─────────────────────────────────────────────────────────────┤
│  数据层（torrent_sources）— 工具轨 T0                        │
│  四层数据源聚合 → 归一化 → D1 上线                           │
│  目标：只存元数据，不托管视频                                 │
└─────────────────────────────────────────────────────────────┘
```

**v1.1 核心策略：先工具、后内容。**

| 原则 | 含义 |
|------|------|
| 工具先行 | 数据管道、IG 引擎、测速、页面生成器在批量上线页面前完成 |
| 内容延后 | GSC 提交与规模扩展推迟到工具能自动产出 IG 信号之后 |
| 沙盒利用 | M1~M3 不增页期间专注工具轨，不浪费 crawl budget |
| 生成器统一 | 验证集 20 页与后续 10K 页共用同一套「槽位 → HTML」逻辑 |

**结论：** 成功的关键不在「有多少 magnet」，而在「每个 magnet 附带多少独有分析」——且这些分析必须由工具自动写入页面，而非事后手工回填。

### 1.1 与竞品定位对照表

下表将「下载站」与「Release 导航站」的差异落到**可判定的维度**。排期争论、页面验收、GSC 提交门槛，均以此表为准——凡落入左列做法的，默认 **不做或延后**。

| 维度 | 典型 magnet 聚合站（竞品） | ReleaseMatch（本站） | 可执行决策 |
|------|---------------------------|----------------------|------------|
| **产品心智** | 「这集有多少个下载链接」 | 「这集该下哪个 release、为什么」 | 功能描述以 **Release 决策** 为主语，不以 magnet 数量为主语 |
| **页面核心单元** | magnet 行（文件名 + size + seeders） | **槽位**（`tmdb_id + season + episode`）+ Recommended Release 卡片 | 一页 = 一槽位；列表表是辅助，推荐卡是主角 |
| **Information Gain 来源** | 无或仅 quality 标签；与 TOP10 页 magnet 同源 | Recommended Release、对版说明、Group 信誉、编码分析、测速（T1~T2） | 无 IG 模块的页 **不得 index**（薄页门禁 ≥2 magnet + IG 区块） |
| **内容建设节奏** | 先铺页抢词，后补数据/文案 | **先工具后内容**：T0→T3 闭环后再 C1/C2 | GSC 提交仅在 **C2**；T3 前禁止批量上线可索引页 |
| **页面生产方式** | 手工模板 / 爬虫洗稿 / AI 批量文案 | T3 生成器：`槽位 JSON → 带 IG 的静态 HTML` | C1 验证集 20 页与 C4 的 10K 页 **共用同一生成器** |
| **数据与托管** | 常混同「提供下载」；部分代下、托管 | **只存元数据**，不托管视频、不代下载 | `torrent_sources` 仅索引；出站 magnet `rel="nofollow"` |
| **域名与生态** | 单站 `/download/` 或站群互链灌权重 | **独立顶级域名**；字幕主域仅单集 **1 条** 出站链 | 禁止主域 `/download/`、禁止两域 sitewide 互链 |
| **SEO 规模策略** | 6 万+ 薄页、长尾词铺量 | M12 **5K~10K 页**，冷启动分批；M1~M3 沙盒期 **不增页** | 收录率低于 25% 时不启动 C4 规模扩展 |
| **成功指标** | 页数、magnet 条数、日抓取量 | **每页 IG 信号数**、Recommended 覆盖率、测速覆盖率 | 验收看 JSON/HTML 是否含 `is_recommended`、`recommend_reason`，而非 magnet 计数 |

**竞品页 vs 本站页（同槽位 Breaking Bad S04E06，示意）：**

```
竞品（典型 magnet 聚合站）：
  Breaking.Bad.S04E06.1080p.WEB-DL.GROUP-X   2.4 GB   24 seeders
  Breaking.Bad.S04E06.720p.WEB-DL.GROUP-Y    1.1 GB   18 seeders
  ← 只有清单，无推荐、无对版依据

本站（Release 导航站）：
  ⭐ Recommended: GROUP-X  1080p WEB-DL  2.4 GB  24 seeders
     理由：Scene 合规；Amazon Prime US WEB-DL；DDP5.1 经校验无延迟
     Group badge：A 级 · 跨源 3/3 一致 · 测速 4.2 MB/s（T2）
  All Sources（8 matched）…
  ← 多 3~5 条竞品页无法拼凑的增量信息
```

### 1.2 定位自检清单（排期 / 验收 / 上线前六问）

任一功能、页面或 PR 合并前，须能勾选 **全部** 「是」；任一为「否」则按左列（竞品）路径处理，**退回工具轨或延后内容轨**。

| # | 自检项 | 是 / 否 | 不通过时的动作 |
|---|--------|---------|----------------|
| 1 | 该产出是否帮助用户 **选定一个 Release**，而非仅多列 magnet？ | | 补 T1 Recommended / 对版模块 |
| 2 | 页面上是否有 **至少 1 块** 竞品 TOP10 没有的独有信息（非 TMDB 复述）？ | | 不得 index；留在 C0 壳或 noindex |
| 3 | IG 内容是否由 **工具自动写入**（`scorer.py` / 生成器），而非手工事后填？ | | 先完成 T3 生成器或 T1 引擎 |
| 4 | 是否遵守 **≥2 magnet** 才允许 index 的薄页门禁？ | | 槽位数据不足则 404/410 或 noindex |
| 5 | 是否在 **T3 页面生成器** 就绪前提交 GSC / 批量扩页？ | | 延后至 C2；当前只做 C0 Trust 壳 |
| 6 | 是否涉及代下载、主域下载路由、或 sitewide 跨站互链？ | | 直接否决（见 §2.5 明确不做） |

**轨道归属速查：** 凡增强「单槽位 Release 决策质量」的 → **工具轨 T**；凡扩大「可被收录的槽位页数量」的 → **内容轨 C**，且 C 不得领先于 T3。

## 三、原方案的结构性问题

### 问题 1：内容与工具混在同一优先级层

原 R0 将「Demo 50 页 + GSC 提交」与「torrent_sources MVP」并列 Week 1~2，导致 Google 可能在 IG 引擎就绪前抓取纯 magnet 列表页。

### 问题 2：Recommended Release 未单列开发项

文档第一 IG 来源长期停留在页面模板描述，缺少 `recommended/scorer.py` 级实现任务（已在 releasematch 脚手架中补齐）。

### 问题 3：页面手工建设 vs 规模扩展

手工 Demo 50 页与后续 +200/批扩展逻辑不统一；**页面生成器（T3）** 应早于验证集（C1）。

## 九、总结

```
工具轨（先做）：
  T0  数据能拉（torrent_sources MVP）
  T1  能推荐（Recommended + Group + 编码 + 跨源）
  T2  能实测（测速 P1 + seeders 日更）
  T3  能出页（页面生成器 + D1 sync）    ← 工具链闭环
  T4  能外延（Stremio > 扩展 > Community API）
  T5  能加深（测速 P2 + 编码 P2 + 日韩/趋势）

内容轨（后做）：
  C0  合规壳（Trust，不 GSC）
  C1  验证集 20 页（生成器第一次运行）
  C2  GSC 提交（首次抓取即完整 IG）
  C3  沙盒观察（不增页，并行 T4）
  C4  规模扩展（收录率门槛 + 生成器 batch）

X   永远不做：代下载、工具完成前批量薄页
```
