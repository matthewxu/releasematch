## 一、总览：三层分工

ReleaseMatch 线上架构可拆成三层，**不要混为一谈**：

```
┌──────────────────────────────────────────────────────────────┐
│  数据层     MySQL（本地） / D1（生产，可选）                    │
│             pipeline 写入 recommended、sources、seeders 等     │
└────────────────────────────┬─────────────────────────────────┘
                             │ 读库
                             ▼
┌──────────────────────────────────────────────────────────────┐
│  生成层     portal/generator（Python + Jinja2）                │
│             generate page / generate all → portal/dist/        │
│             serve 仅本地开发预览，非生产方案                     │
└────────────────────────────┬─────────────────────────────────┘
                             │ 上传 dist
                             ▼
┌──────────────────────────────────────────────────────────────┐
│  托管层     Cloudflare Pages 或 Workers Static Assets          │
│             向访客提供已生成的静态 HTML（访问时不查库）           │
└──────────────────────────────────────────────────────────────┘
```

**一句话：** 用户打开 episode 页时读的是 **HTML 文件**，不是实时查 D1；D1/MySQL 是 **写数据 + 生成 HTML 时用**。

## 二、常见问题速查

| # | 问题 | 简短结论 |
|---|------|----------|
| Q1 | 静态页上 Cloudflare Pages 后，还要 D1 吗？ | **展示页面不必**；**API / 云端 pipeline / 无本机 MySQL 时建议要** |
| Q2 | 要不要自建 VPS 专门 generate？ | **不要**；本机或 CI generate，再 deploy |
| Q3 | Cloudflare 能用 Worker 在线 `generate all` 吗？ | **不能指望单次 Worker 批量跑完**；用 **CI build** 或 **Queue 分批** |
| Q4 | `serve` 能当生产方案吗？ | **不能**；仅本地读 MySQL 动态渲染 |
| Q5 | 手写 `portal/breaking-bad/...` 和 generator 啥关系？ | **已删除**（2026-07-05）；内容仅来自 DB → `portal/dist/` |
| Q6 | Pages Free 够放多少页？ | 约 **2 万文件/站点**；M12 目标 5K–10K 页足够 |

以下逐条展开。

## 三、Q1 — 上了静态 Pages，是否就不需要 D1？

### 3.1 结论

| 场景 | 是否需要 D1 |
|------|-------------|
| 只做 **静态 SEO 站点**（Recommended、Sources 已 bake 进 HTML） | **可以不要** |
| **Stremio / Workers API**（如 `/api/v1/sources?tmdb=…`） | **需要**（或等价线上库） |
| **云端 / CI 生成**，开发机没有 MySQL | **需要 D1**（或 R2 导出 JSON 等） |
| 本机 MySQL → `generate all` → deploy，近期无 API | **可以暂缓 D1** |

### 3.2 为何项目仍保留 D1 设计？

D1 在规划里是 **线上数据仓库**，不是给每个 PV 查库：

| 用途 | 访客打开静态页时要 D1 吗 |
|------|---------------------------|
| 看 `/breaking-bad/s4e6/` 页面内容 | 否 |
| pipeline 更新 magnet / seeders 后再生成 | 是（或 MySQL + 本机 generate） |
| T4 Stremio 插件读运行时 API | 是 |
| MySQL → 生产 sync（T3 Worker，待建） | 是 |

### 3.3 两条路线

**路线 A — 极简（C0 / C1 推荐）**

```
本地 MySQL → pipeline → generate all → portal/dist → wrangler pages deploy
```

- 不上 D1、不上 VPS
- 更新数据：本机（或 CI 能连 MySQL）跑 pipeline + generate + 再部署

**路线 B — 完整（T3 / T4 / C4）**

```
pipeline → sync → D1 →（CI 或 Queue 触发）generate → Pages
                          ↓
                   Workers API（Stremio 等）
```

- D1 为生产数据源；MySQL 仍为本地开发与测试

## 四、Q2 — 静态页要不要单独服务器生成？

### 4.1 结论

**不需要 VPS。** 生成发生在 **开发者机器** 或 **CI（GitHub Actions / Pages Build）**，产物是 `portal/dist/` 里的 HTML；Cloudflare 只负责 **托管静态文件**。

### 4.2 推荐流程

```bash
cd releasematch

# 1. 数据就绪（MySQL 或未来 D1）
python -m workflow.run pipeline slot --tmdb-id 1396 --season 4 --episode 6

# 2. 生成静态页
python -m workflow.run generate all
# 或单页：python -m workflow.run generate page --path /breaking-bad/s4e6/

# 3. 部署（Direct Upload，无需 Git 集成亦可）
npx wrangler pages deploy portal/dist --project-name=releasematch
```

### 4.3 本地预览（非生产）

```bash
python -m workflow.run serve --port 8765
```

- 读 MySQL **动态渲染**，方便改模板
- **不要** 当作线上架构；生产用 `generate` + deploy

## 五、Q3 — Cloudflare 支持 Worker 在线 generate Pages 吗？

### 5.1 直接回答

**部分支持，但不是「一个 Worker 在边缘跑完 Python `generate all` 再自动出站」。**

Cloudflare 常见做法是：

| 方式 | generate 发生在哪里 | 适合 ReleaseMatch？ |
|------|---------------------|---------------------|
| **Pages / GitHub Actions 构建** | Build 容器 / Runner | ✅ **最推荐** |
| **`wrangler pages deploy dist`** | 仅上传已有 HTML | ✅ **推荐** |
| **Worker 按需 SSR**（读 D1 现渲染） | 每次请求 1 页 | ⚠️ 与「预生成 SEO 静态页」路线不同 |
| **Cron + Queue + Worker 分批** | 每消息 1 页或 1 批 | ⚠️ 可行，需重写生成器为 TS/JS |
| **单次 Worker 跑完 generate all** | Worker 内 | ❌ **不现实** |

### 5.2 Worker 不适合 bulk generate 的原因

| 限制 | 说明 |
|------|------|
| CPU 时间 | Free 约 10ms；Paid HTTP 默认 30s，最高约 5 分钟 |
| 内存 | 128 MB |
| 静态文件数 | 每 Worker 版本最多 **20,000** 个文件 |
| 技术栈 | 当前生成器为 **Python + Jinja2 + mysql_store**，不能原样搬进 Worker |

5K–10K 页批量渲染必须在 **CI** 或 **Queue 分批任务** 中完成。

### 5.3 三种「线上 generate」形态对比

#### 形态 1 — CI 构建（推荐）

```
push / cron / workflow_dispatch
  → pip install + python -m workflow.run generate all
  → wrangler pages deploy portal/dist
```

- Worker **不参与** bulk generate
- CI 可读 D1（wrangler / API）或 MySQL（自建 Runner）

#### 形态 2 — Worker 按需 SSR

```
GET /breaking-bad/s4e6/ → Worker → 查 D1 → 模板 → HTML
```

- 完全在线生成，但每 PV 消耗算力
- 不适合 M12 大规模 SEO 静态页策略

#### 形态 3 — Queue 分批 + R2 / Direct Upload（进阶）

```
Cron → Queue（每消息 1 页）→ Consumer 读 D1 → 渲染 → 写 R2
  → 定期 wrangler deploy 或 Pages Deployments API
```

- Worker **参与** generate，但需 **重写** 生成逻辑并设计部署流水线
- 复杂度高于形态 1

### 5.4 Workers Static Assets（2025+ 方向）

Cloudflare 新站更推荐 **Worker + `[assets]`** 托管静态目录（Pages 新功能放缓，存量 Pages 仍可用）：

```jsonc
// wrangler.jsonc 示例
{
  "name": "releasematch",
  "compatibility_date": "2026-06-29",
  "assets": {
    "directory": "./portal/dist",
    "html_handling": "force-trailing-slash"
  }
}
```

部署命令：`npx wrangler deploy` — 仍是 **先 generate，再 upload**，不是 Worker 运行时批量造 HTML。

### 5.5 自动更新时的分工建议

| 组件 | 职责 |
|------|------|
| **Cron Worker** | sync D1、Webhook 触发 CI |
| **GitHub Actions** | `generate all` + `wrangler pages deploy` |
| **Worker fetch** | Stremio API、非页面类接口 |

**重活 generate 放 CI，Worker 做触发器 + API** — 这是 Cloudflare 生态里更常见的分工。

## 六、Q4 — `serve` 与 `generate` 的区别

| 命令 | 读库 | 输出 | 用途 |
|------|------|------|------|
| `python -m workflow.run serve` | MySQL 实时 | HTTP 动态 HTML | **本地开发预览** |
| `python -m workflow.run generate page/all` | MySQL | `portal/dist/` 静态文件 | **生产部署前生成**（`all` 含 Trust + static_shell） |
| `python -m workflow.run serve-static` | `portal/dist/` | 纯静态 HTTP | **dist 部署同源预览**（自动 sync static，内联 i18n） |
| `python -m http.server` 在 `portal/dist/` | 无 | 仅当已 `generate all` | 可用但不推荐；优先 `serve-static` |

## 七、Q5 — Cloudflare Pages 容量与成本（静态站）

| 项 | Free 档参考 | ReleaseMatch 影响 |
|----|-------------|-------------------|
| 静态文件数 / 站点 | 约 20,000 | M12 目标 5K–10K 页，有余量 |
| 带宽 | 静态一般不限于小站场景 | SEO 冷启动阶段通常够用 |
| 单次 Direct Upload manifest | 最多 20,000 条目 | 超量需分批 deployment 或 Workers Paid 档 |

详细配额以 [Cloudflare 官方文档](https://developers.cloudflare.com/workers/platform/limits/) 为准。

## 八、阶段建议（对齐 T / C 双轨）

| 阶段 | 数据库 | 生成 | 托管 | D1 |
|------|--------|------|------|-----|
| **C0 / C1**（Trust 壳 + 小集验证） | MySQL | 本机 `generate` | Pages Direct Upload | **可不上** |
| **C2 / C3**（扩页 + 模板稳定） | MySQL → 可选 sync | CI `generate all` | Pages | 可选 |
| **T4**（Stremio API） | D1 | CI 或 SSR API | Pages + Worker | **需要** |
| **C4**（5K–10K 页 + 定期更新 seeders） | D1 + sync | CI 定时 generate | Pages | **强烈建议** |

## 九、决策树（快速选型）

```
需要 Stremio / 运行时 JSON API？
├─ 是 → 上 D1 + Worker API；页面仍建议 CI generate 静态 HTML
└─ 否 → 仅静态 SEO 站？
         ├─ 是 → MySQL + 本机/CI generate → Pages（D1 可延后）
         └─ 否 → 重新评估产品范围

generate 跑在哪里？
├─ 有 Python 环境（本机 / Actions）→ generate all + wrangler deploy ✅
├─ 只有 Cloudflare、无 CI → 需 D1 + Worker SSR 或 Queue 分批（成本高）
└─ 想 Worker 一次跑完 10K 页 → ❌ 不可行

本地怎么预览 DB 真实数据？
└─ workflow.run serve 或 generate page 后打开 portal/dist/
```
