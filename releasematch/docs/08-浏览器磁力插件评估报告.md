## 一、开发时间评估

### 1.1 模块拆分（Manifest V3 版）

| 模块 | 估时 | 说明 |
|------|------|------|
| MV3 框架搭建（manifest + Service Worker） | **2 天** | `declarativeNetRequest` 替代 webRequest 拦截；Service Worker 生命周期管理比 V2 复杂 |
| qBittorrent Web API 集成（v4/v5 双兼容） | **2 天** | qBittorrent v5（2025 年底发布）API 有 breaking change，需同时兼容两版 |
| magnet 协议拦截 + 弹窗 UI | **1.5 天** | MV3 下 `chrome.webNavigation.onBeforeNavigate` 无法可靠拦截 `magnet:` 协议，需内容脚本嗅探 + background 双保险 |
| 本站推荐 API 集成（核心差异） | **2 天** | **新增**——这是与 Magnetar 的核心差异：扩展内嵌 Recommended Release + 跨源 badge |
| 测速数据回传模块 | **1 天** | Opt-in 控制、API 通信、去重、频率限制 |
| 隐私设计 + Opt-in 流程 | **1 天** | 首次运行弹窗、Options 页面、数据最小化策略 |
| 中英文双语 UI | **0.5 天** | i18n messages.json 两套 |
| 打包 + 商店发布（Chrome） | **1 天** | 图标、截图、隐私政策、CWS 审核准备 |
| Firefox 适配 + 发布 | **0.5 天** | 审查 API 差异（无 $5 注册费） |
| 测试 + 调试 | **1.5 天** | 跨浏览器、qBittorrent v4/v5、无 qB 时的 Graceful Degradation |
| **合计** | **~11 天** | 可与人天并行 |

### 1.2 与文档原估算的差异

文档原估算 **7 天**。差额 **+4 天**，原因：

1. **MV3 下 magnet 拦截不可靠**（原文档假设 V2 行为）
2. **qBittorrent v5 兼容** 增加复杂度（2025 年底前文档未覆盖）
3. **推荐 API 集成** 是新需求——如果只做纯转发器，不需要这 2 天

### 1.3 建议 MVP 范围

```
v0.1 — 不做：
  ❌ 批量操作（Magnetar 有，但非核心）
  ❌ 云后端（Real-Debrid/TorBox 等）
  ❌ 自建 Native Host
  ❌ Firefox 同步发布（延后 1 周）

v0.1 — 必须做：
  ✅ Manifest V3
  ✅ magnet 协议拦截 → qBittorrent Web API（v4/v5 兼容）
  ✅ 本站推荐 API → 显示 Recommended Release + 跨源 badge
  ✅ 测速数据回传（opt-in）
  ✅ 中英文 UI
  ✅ 开源（MIT）
```

### 2.2 关键结论

全球有 **1.7 亿 P2P 用户**，其中 uTorrent Web 的 **1,380 万 MAU** 是最直接的目标池——这些用户已经习惯在浏览器环境中处理 torrent 操作。这是一个巨大的存量市场，且现有的 magnet 转发器工具均未与 release 推荐引擎深度绑定。

## 三、竞品格局

### 3.1 直接竞品

| 竞品 | 开发状态 | 用户量级 | 功能 | 核心缺陷 |
|------|---------|---------|------|---------|
| **Magnetar** | ✅ 活跃，开源 MIT | 估计 **50K~200K** 安装 | 6 种后端、多语言 11 种、批量操作、缓存检测 | 纯工具，不提供 release 推荐，不对齐任一内容站 |
| **Remote Torrent Adder** | ⚠️ MV2→MV3 重构中 | 曾 ~100K，现已下滑 | 支持最多客户端的通用适配器 | 老代码，V3 版切割了旧客户端 |
| **Magnet2Deluge** | ⚠️ 低维护 | **~750** 安装 | 仅 Deluge | 过于专一 |
| **Add Remote Torrent** | ✅ 活跃 | **~10K~50K** 安装 | 多 server profile | 无云后端，纯 WebUI 转发 |

### 3.2 市场格局判断

- **通用磁力转发器** 是饱和但分散的市场。Magnetar 领先但未垄断——CWS 上搜索 "magnet" 有 ~40 个扩展，没有一家超过 20 万用户
- **垂直品类（推荐 + 集成）** 是空白。没有任何扩展能告诉你「这个 release 是否被多源交叉验证过」或「当前页面的 Recommended 版本是哪个」
- **中国区市场** 是明显空缺。现有扩展均以英文为主，无中文 UI、无中文下载站适配

### 3.3 与 Magnetar 的直接对比

```
                    Magnetar                 ReleaseMatch Extension
                    ────────                 ─────────────────────
核心功能            magnet → 任意后端         magnet → 后端 + 推荐决策
推荐引擎            ❌ 无                    ✅ 本站 scorer + cross-source
跨源验证            ❌ 无                    ✅ cross_source_count badge
Group 信誉           ❌ 无                    ✅ groups.yaml L0~L4 标注
测速数据            ❌ 无（仅转发）           ✅ VPS + 社区 opt-in 回传
云后端              ✅ 6 种                   ❌ 计划中（未来）v1 不做
本地客户端           ✅ qBittorrent 等         ✅ qBittorrent v4/v5
批量操作             ✅                       ❌ v1 不做
语言               11 种英文主导             中英文双语
隐私                ✅ 无遥测                 ✅ Opt-in 透明回传
开源               ✅ MIT                    ✅ MIT
```

### 3.4 竞争壁垒

Magnetar 最怕的不是另一个转发器——最怕的是 **「你的扩展帮用户做下载决策」**。功能复制容易，但 cross-source database + scorer + groups registry 这套数据管线需要：

- 四层数据源管道（已建成）
- 推荐引擎规则（已建成）
- 持续维护的 Group 信誉数据库（进行中）
- 用户测速回传网络效应（未成形）

**执行窗口：** Magnetar 目前是 1 人开源项目（arrcee），如果它半年后加上推荐功能，窗口就关了。因此扩展的差异化功能（与本站 API 集成）**必须与扩展同时发布**，不能先发纯转发器再补推荐。

## 四、用户价值评估

### 4.1 解决的问题

| 用户痛点 | 现有方案 | ReleaseMatch 扩展的解决 | 价值强度 |
|---------|---------|----------------------|---------|
| 点 magnet 链接后浏览器弹窗确认 → 慢 | 手动确认 | 一键发送 | ⭐⭐⭐ |
| 几十个 release 不知道下哪个 | 无方案 / 凭感觉 | 扩展内直接显示 **本站 Recommended** + 跨源 badge | ⭐⭐⭐⭐⭐ |
| 下载完才发现画质/码率不对 | 无方案 | 页面已标注，扩展同步展示 | ⭐⭐⭐⭐ |
| 在不同下载站之间切来切去比较 release | 开多个 tab 手动对比 | 扩展内聚合本槽位所有源的核心差异 | ⭐⭐⭐⭐ |
| 想贡献测速数据但怕隐私泄露 | 无方案 | Opt-in + 透明源码 + 最小化采集 | ⭐⭐⭐ |

### 4.2 核心价值主张

> 不是「又一个 magnet 转发器」，而是 **「带推荐引擎的下载决策辅助工具」**。

### 4.3 用户画像

```
Primary: 欧美 torrent 下载老手
  年龄 22~40，男性为主
  已安装 qBittorrent/Deluge/Transmission
  每周下载 3~15 个 torrent
  痛点：从下载站点开 magnet → 弹窗确认 → 跳转客户端 → 几十个里选哪个
  当前行为：手动复制 magnet / 用 Magnetar 转发

Secondary: 日韩/中文区用户
  使用 Nyaa / 动漫站 / 日剧站
  痛点：中文/日文标题在通用扩展里显示为乱码
  当前行为：手动下载 .torrent 文件

Tertiary: Stremio 用户
  用插件浏览但不直接下载
  扩展是「离线 fallback」场景
```

## 五、对 ReleaseMatch 的回馈价值

### 5.1 五重回馈

| # | 价值 | 说明 | 与 IG 的关系 |
|---|------|------|------------|
| 1 | **用户侧测速数据** | 数千用户不同地区/ISP 的真实下载速度 | IG 最高级（独家聚合数据） |
| 2 | **Chrome Web Store 外链** | CWS 页面为 dofollow 链接 | 高质量外链 + 品牌搜索 |
| 3 | **GitHub 仓库自然外链** | 开源扩展的 README → 站内链 | 开发者社区信任 |
| 4 | **安装量 = 品牌曝光** | 每次用户从 CWS 搜索发现扩展 | 品牌词搜索量增长 |
| 5 | **直接流量入口** | 扩展内「Visit Website」按钮 | 直接 UV 增长（跳过搜索） |

### 5.2 测速数据对比

| 维度 | VPS 测速 | 扩展测速 |
|------|---------|---------|
| 节点 | 单节点（VPS 机房） | 多地区/多 ISP |
| 每日数据量 | ~600 次 | M6 后预计 **450 次/天** |
| 数据一致性 | 高（可控） | 中（opt-in，质量参差） |
| IG 价值 | 高 | **最高**（独家聚合 + 多视角） |
| M12 累积 | 日 600 次 | 日 1,350 次 |

VPS 与扩展是 **互补关系**，不是替代关系。扩展是测速 IG 从「高」到「最高」的必经之路。

## 六、战略定位与执行建议

### 6.1 与 Stremio 插件的先后顺序

| 维度 | Stremio 插件 | 浏览器扩展 |
|------|------------|-----------|
| 开发时间 | ~5 天 | ~11 天 |
| 用户决策门槛 | 低（安装即用） | 中（需配置 qBittorrent） |
| 外链价值 | 中（GitHub） | 高（CWS dofollow） |
| 测速数据回馈 | 无 | **核心用途** |
| 品牌曝光 | 中 | 高 |
| 直接 UV | 低 | 中 |

**结论：Stremio 优先发布**——开发量小，5 天 MVP，先拿到用户反馈和外链。扩展的推荐 API 与 Stremio 插件共享后端设计，两个工具之间的技术栈复用度高。

### 6.2 建议发布路径

```
Week 1        Stremio 插件 v0.1 → GitHub + Reddit r/StremioAddons

Week 2~3      扩展开发（11 天）
              ├── Day 1~2: MV3 框架搭建
              ├── Day 3~4: qBittorrent v4/v5 双 API 集成
              ├── Day 5~6: magnet 拦截 + 弹窗 UI
              ├── Day 7~8: 本站推荐 API 集成（核心差异）
              ├── Day 9:   测速回传 + Opt-in 流程
              ├── Day 10:  测试 + 中英文双语
              └── Day 11:  打包 + 提交 CWS

Week 4        CWS 审核（1~3 工作日）
              审核期间 → Firefox 适配 + 提交 Firefox Add-ons

Week 5+       Reddit r/torrents + r/Usenet 发布
              站内推广（页面 Banner + 推荐模块旁引导安装）
```

### 6.3 止损条件

如果以下任一条成立，应重新评估或暂停：

1. CWS 审核因内容政策拒绝（准备 DMCA 政策链接 + 避免 "pirate" 表述）
2. Magnetar 先发布了推荐功能
3. 发布后 3 个月安装量 < 500
4. qBittorrent 放弃 Web API（可能性极低）

## 七、风险与最坏情况评估

| 风险 | 概率 | 影响 | 应对 |
|------|------|------|------|
| CWS 审核拒绝（内容政策） | **中** | 高 | 确保不含 "pirate/download movies free"；用 "release navigation + magnet organizer" 描述；准备 DMCA 政策链接 |
| Magnetar 先加推荐功能 | **低~中** | 中 | 数据管道（四源 + scorer）深于单靠 CWS 爬虫；但不可掉以轻心 |
| qBittorrent v5 覆盖率低 | **低** | 低 | v1 同时兼容 v4/v5 |
| 安装量达不到预估 | **中** | 低 | 核心价值不在安装数，在测速数据——即使 1K 安装，每天 ~30 次测速也能补充 VPS 盲区 |
| 被标记为 "deceptive" | **低** | 高 | 明确标注功能、透明代码、不上传用户数据 |

## 八、结论

### 8.1 做还是不做

**做，但精确限定范围。**

| 问题 | 答案 |
|------|------|
| 该不该做 | ✅ **应该做。** 1.7 亿 P2P 用户、5,400 万月活是基本盘；差异化推荐能力是 Magnetar 等竞品没有的护城河 |
| 什么时候做 | **Stremio 插件发布后立即启动**。预计 C2（Week 9~10）附近 |
| 花多少时间 | **11 天 MVP**，v1 不做云后端、不做批量、只发布 Chrome |
| 预期效果 | M12 累计安装 10K~30K，每日测速数据 300+ 条，CWS dofollow + 品牌搜索双增益 |
| 不做会怎样 | VPS 测速始终单节点盲区，社区测速飞轮无法启动，Magnetar 巩固市场后更难切入 |

### 8.2 一句话总结

> 扩展不是锦上添花——是测速 IG 从「高」到「最高」的必经之路，也是品牌从「搜索依赖」到「直接触达」的跨越。前提是不能做成又一个 Magnetar，**必须把推荐引擎内嵌作为核心差异**。
